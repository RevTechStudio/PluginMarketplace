# SDD Help - ソフトウェア開発ワークフロー

このヘルプでは、RevTechStudio SDDプラグインの全コマンドと使用手順を説明します。

## 概要

SDDプラグインは、プロジェクトの分析から実装、品質保証までの一連のソフトウェア開発プロセスを支援します。
すべての成果物は `./.spec/` ディレクトリに保存されます。

## ワークフロー

```
/analyze → /brainstorm → /requirement → /design → /tasks → /implement → /check-discrepancies
```

各コマンドは前のコマンドの出力を入力として使用し、段階的にプロジェクトを進めていきます。

---

## コマンド一覧

### 1. `/analyze` - プロジェクト分析

**目的**: プロジェクトの構造と技術スタックを分析し、ドキュメント化する

**入力**: なし（現在のプロジェクトを分析）

**出力**: `./.spec/project-analysis.md`

**内容**:
- プロジェクトの構造把握
- 使用技術の特定
- 既存のアーキテクチャ分析

**実行方法**:
```
/analyze
```

**次のステップ**: `/brainstorm` でアイデアを深掘りする

---

### 2. `/brainstorm` - アイデアの深掘り

**目的**: Why/Who/What の3軸でプロジェクトアイデアを深掘りする

**入力**: `./.spec/project-analysis.md`

**出力**: `./.spec/idea.md`

**内容**:
- Why（なぜ）: プロジェクトの目的と価値
- Who（誰が）: ターゲットユーザー
- What（何を）: 提供する機能とサービス
- 類似サービスの調査
- 建設的な批判とピボット提案

**実行方法**:
```
/brainstorm
```

**次のステップ**: `/requirement` で要件定義を行う

---

### 3. `/requirement` - 要件定義

**目的**: アイデアを実装可能な要件定義に変換する

**入力**: `./.spec/idea.md`

**出力**: `./.spec/requirements.md`

**内容**:
- 機能要件の抽出
- 非機能要件の定義（性能、セキュリティ、可用性、保守性、互換性）
- ユーザーストーリーの作成
- 優先度設定（P1-P4）

**実行方法**:
```
/requirement
```

**次のステップ**: `/design` で詳細設計を行う

---

### 4. `/design` - 詳細設計

**目的**: 要件から実装可能な詳細設計を作成する

**入力**: `./.spec/requirements.md`

**出力**: `./.spec/design.md`

**内容**:
- 技術スタックとフレームワークの選定
- システムアーキテクチャ設計
- データベース設計（ER図、テーブル、インデックス）
- API設計（エンドポイント、リクエスト/レスポンス形式）
- セキュリティ実装設計
- ディレクトリ構造の提案

**実行方法**:
```
/design
```

**次のステップ**: `/tasks` でタスク計画を作成する

---

### 5. `/tasks` - タスク計画

**目的**: 設計を実装可能なタスク計画に変換する

**入力**: `./.spec/design.md`

**出力**: `./.spec/tasks.md`

**内容**:
- 開発環境の確認
- 実装優先順位と順序の決定
- 実装可能なタスクへの分割
- タスクの依存関係の明確化

**実行方法**:
```
/tasks
```

**次のステップ**: `/implement` で実装を実行する

---

### 6. `/implement` - 実装

**目的**: タスク計画に基づいてコードを実装する

**入力**: `./.spec/tasks.md`

**出力**: 生成されたコードファイル

**内容**:
- タスク計画に沿ったコンポーネント/モジュールの実装
- テストコードの作成
- 品質チェック（linter、formatter）
- ドキュメントの更新
- Git操作（規約に準拠）
- 実装レポートの作成

**実行方法**:
```
/implement
```

**次のステップ**: `/check-discrepancies` で整合性を検証する

---

### 7. `/check-discrepancies` - 整合性検証

**目的**: すべてのドキュメントとコードの整合性を検証する

**入力**: すべての生成ファイル

**出力**: 矛盾点のレポート

**内容**:
- ドキュメント間の整合性チェック
- ドキュメントとコードの整合性チェック
- 規約への準拠確認
- 設定ファイルの整合性チェック

**実行方法**:
```
/check-discrepancies
```

## 使用例

### 新規プロジェクトの場合

```bash
# 1. プロジェクトを分析
/analyze

# 2. アイデアを深掘り
/brainstorm

# 3. 要件を定義
/requirement

# 4. 詳細設計を作成
/design

# 5. タスクに分割
/tasks

# 6. 実装を実行
/implement

# 7. 整合性を検証
/check-discrepancies
```

### 既存プロジェクトに機能追加する場合

```bash
# 既存の分析がある場合は /brainstorm から開始可能
/brainstorm

# または既存の要件がある場合は /design から開始可能
/design

# 以降は通常のワークフローに従う
/tasks
/implement
/check-discrepancies
```

## 出力ファイル構造

すべての成果物は `./.spec/` ディレクトリに保存されます：

```
.spec/
├── project-analysis.md    # /analyze の出力
├── idea.md                # /brainstorm の出力
├── requirements.md        # /requirement の出力
├── design.md              # /design の出力
└── tasks.md               # /tasks の出力
```

---

## 規約への準拠

すべてのコマンドは以下の規約に準拠します：

- **document-conventions**: ドキュメント作成の標準
- **development-conventions**: 開発プラクティス
- **coding-conventions**: コーディング規約
- **branch-commit-conventions**: Git操作の規約

これらの規約は `rev-tech-studio-foundations` プラグインで定義されています。

---

## ヒント

1. **順序を守る**: 各コマンドは前のコマンドの出力に依存しています
2. **柔軟な開始点**: プロジェクトの状況に応じて途中のコマンドから始めることも可能
3. **定期的な検証**: `/check-discrepancies` で定期的に整合性を確認しましょう
4. **反復的な改善**: 必要に応じて前のステップに戻って修正することも可能

---

さらに詳しい情報が必要な場合は、各コマンドを実行して詳細な手順を確認してください。

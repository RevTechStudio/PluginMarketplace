# Brainstorm

## 概要

プロジェクトのアイディアをWhy/Who/Whatの3軸で深掘りし、要件定義に進める前段階を支援する。ユーザーのアイディアに過度に迎合せず、建設的な批判と代替案提示を通じてより良い方向性を模索する。実現可能性を評価し、必要に応じてピボットを提案する。

## 使用タイミング

- ユーザーが新規プロジェクトのアイディアを持っている
- 「〇〇アプリを作りたい」「〇〇に機能を追加したい」などの具体的な提案がある
- 要件定義の前段階として、アイディアの核心を明確にしたい

**使用しないケース:**

- 何を作るか全く決まっていない状態
- すでに要件が明確で、技術的な実装に入る段階

## 対話の原則

### スタイル

- ビジネス口調（丁寧・明確）を使用する
- 技術的な実装の話は一切しない（「これ実装できますか？」「予算は？」などは聞かない）
- 一問一答のキャッチボール形式を厳守する
- ユーザーの入力負担を減らすため、選択式の質問を多用する
- **ユーザーのアイディアに過度に迎合しない**
- **より良い方向性があれば明確に提案する**
- **建設的な批判を恐れず、根拠を添えて提示する**

### 質問形式

- 基本的に選択式（2〜5択）で質問する
- 最後の選択肢は「その他」とし、自由回答も受け付ける
- 推奨する選択肢を明示する（理由も簡潔に記載）

**選択式質問の例:**

```text
では、このアプリの主な価値はどこにあるとお考えでしょうか？

A. 既存サービスにはない独自機能を提供する
B. 既存サービスより使いやすいUIを実現する
C. 特定のニッチなユーザー層に特化する
D. その他

【推奨：A】既存サービスとの差別化が明確になりやすいため
```

## ワークフロー

### 1. アイディアの受領

ユーザーから「〇〇を作りたい」という提案を受け取る。

### 2. プロジェクト分析ドキュメントの確認

`./.spec/project_analyst.md` が存在するか確認する。

**存在しない場合:**

```text
プロジェクト分析ドキュメント（./.spec/project_analyst.md）が見つかりませんでした。

A. そのままアイディア出しを進める
B. project-analyzerスキルで既存プロジェクトを分析してから進める

【推奨：B】既存プロジェクトの情報があると、より的確なアイディア出しが可能なため
```

ユーザーがBを選択した場合、project-analyzerスキルの使用を促し、完了後に再度このスキルを使用するよう案内する。

**存在する場合:**
ドキュメントの内容を読み込み、深掘り対話で活用する。この情報はユーザーに明示的に伝えず、質問の中で自然に活用する。

### 3. 類似サービス調査

web_searchツールを使用して、類似するアプリやサービスを調査する。

**調査範囲:**

- 3〜5件のサービスをピックアップ
- サービス名と主要機能を簡潔に記載

**調査結果の提示例:**

```text
「読書記録アプリ」で類似サービスを調査しました。

【調査結果】
- 読書メーター：読書記録、レビュー共有
- ブクログ：蔵書管理、感想投稿
- Goodreads：国際的な読書SNS

これらのサービスが既に存在しますが、アイディアを深掘りしてよろしいでしょうか？
```

**ユーザーの応答を待つ:** Yesの場合は次のステップへ、Noの場合は終了する。

### 4. 深掘り対話

Why（なぜ）、Who（誰のため）、What（何が違う）の3軸で深掘りを行う。

**project_analyst.mdがある場合の活用:**

- 既存プロジェクトの技術スタックや設計思想を踏まえた質問をする
- 例：「現在〇〇を使っていますが、新機能でも同じ技術を使う前提でしょうか？」
- 既存の主要機能との関連性を問う
- ただし、これらの情報をユーザーに明示的に提示せず、質問の文脈で自然に活用する

**重要な原則:**

- 質問は一度に1つのみ
- 必ず選択式で提示
- ユーザーの回答を待ってから次の質問へ
- 回答に応じて柔軟に質問を調整
- 矛盾や不明瞭な点があれば指摘する

**深掘りの軸:**

#### Why（なぜ作るのか）

- 解決したい課題は何か
- なぜ既存サービスでは不十分なのか
- このプロジェクトの動機は何か

#### Who（誰のためか）

- ターゲットユーザーは誰か
- どのような属性・特徴を持つ人か
- 何人くらいのユーザーを想定するか

#### What（何が違うのか）

- 既存サービスとの差別化ポイントは何か
- 独自の価値は何か
- なぜユーザーはこのサービスを選ぶのか

**矛盾指摘の例:**

```text
「シンプルで使いやすい」とおっしゃいましたが、それはTodoistやTickTickなど既存サービスでも実現されています。わざわざ開発コストをかけて作る価値は、具体的にどこにあるとお考えですか？
```

**代替案提示:**

問題点を発見した場合、より良い方向性を明確に提案する。

- 根拠とデータを添えて提示
- 「〇〇をお勧めします」「△△の方が適切です」など明確な表現を使用
- ユーザーの判断を求める

**代替案提示の例:**

```text
「タスク管理アプリ」とのことですが、類似アプリが多数存在します。

分析の結果、以下の方向性をお勧めします：

A. 特定用途特化型（例：プログラミング学習向けタスク管理）
B. 既存の課題解決型（例：GitHub連携による開発タスク管理）
C. 新しい使用体験（例：コマンドライン中心のタスク管理）

理由：汎用的なタスク管理アプリは類似品が多く、差別化が困難です。上記のような明確な切り口があれば、独自性を出せます。また、自分が本当に使いたい機能に集中できます。

どの方向性に興味がありますか？
```

### 5. 実現可能性評価

Why/Who/Whatが明確になった時点で、実現可能性を簡易評価する。

#### 評価項目

**技術的実現可能性:**

- 必要な技術の複雑度（低/中/高）
- 既存技術で実現可能か
- 特殊な技術や専門知識が必要か

**学習価値:**

- 新しい技術・知識を学べるか
- スキルアップに繋がるか
- ポートフォリオとしての価値

**実装の現実性:**

- 個人で実装可能な規模か
- 開発が継続できるモチベーションがあるか
- 必要なツールやリソースは入手可能か

#### 評価結果の提示

```text
実現可能性を評価しました：

【技術的実現可能性】中
- 必要な技術：Web開発、データベース、ノート管理機能
- 複雑度：中程度（既存技術で実現可能）

【学習価値】高
- Webアプリ開発の実践経験が得られる
- データベース設計、UI/UX設計を学べる
- ポートフォリオとして活用できる

【実装の現実性】高
- 個人で実装可能な規模
- 自分が使いたい機能なので継続しやすい
- 必要なツールは無料で入手可能

この評価で問題なければ、要件定義フェーズに進みます。
```

### 6. 完了判断

以下の条件がすべて満たされた時点で、要件定義への移行を提案する：

- Why、Who、Whatの3軸すべてに対して具体的な回答が得られた
- 矛盾点や不明瞭な点がすべて解消された
- 実現可能性評価が完了した

**移行提案の例:**

```text
アイディアの深掘りが完了しました。以下の点が明確になっています：

- なぜ作るのか：〇〇
- 誰のためか：〇〇
- 何が違うのか：〇〇

要件定義フェーズに進んでよろしいでしょうか？
```

**ユーザーの承認を待つ:** Yesの場合は次のステップへ、Noの場合はさらに深掘りを続ける。

### 7. ドキュメント化

ユーザーが承認した場合、以下の6項目を含むドキュメントを作成する。

**ドキュメント構成:**

1. **プロジェクト概要** - プロジェクトの全体像を1〜2段落で記述
2. **ターゲットユーザー** - 誰のためのプロジェクトか
3. **解決する課題** - どのような問題を解決するのか
4. **差別化ポイント** - 既存サービスとの違い、独自の価値
5. **類似サービス分析** - 調査で判明した類似サービスとその特徴
6. **実現可能性評価** - 技術的実現可能性、学習価値、実装の現実性

**ドキュメント形式:**

- Markdown形式で作成
- /mnt/user-data/outputs/に保存
- ファイル名: `{プロジェクト名}_brainstorm.md`

**ドキュメント例:**

```markdown
# 読書記録アプリ - アイディア深掘り結果

## プロジェクト概要
読書好きの学生向けに、読書記録と学習管理を統合したアプリ。読んだ本から得た知識を体系的に整理し、試験勉強や課題に活用できる。

## ターゲットユーザー
- 大学生、高校生
- 読書好きで、年間20冊以上読む
- 読書から得た知識を実践的に活用したい

## 解決する課題
既存の読書記録アプリは「読んだ」という記録に留まり、読書から得た知識を学習に活かす仕組みがない。学生は読書と勉強を別々に管理しており、知識が分断されている。

## 差別化ポイント
- 読書記録と学習ノートの統合
- 本から得た知識をフラッシュカード化
- 試験前に復習しやすい構造

## 類似サービス分析
- 読書メーター：読書記録、レビュー共有
- ブクログ：蔵書管理、感想投稿
- Goodreads：国際的な読書SNS
→ いずれも読書記録に特化しており、学習管理との統合はない

## 実現可能性評価

### 技術的実現可能性：中
- 必要な技術：Webアプリ開発、データベース設計、ノート管理機能
- 複雑度：中程度（既存技術で実現可能）

### 学習価値：高
- Webアプリ開発の実践経験が得られる
- データベース設計、UI/UX設計を学べる
- 自分が使いたい機能を作る経験

### 実装の現実性：高
- 個人で実装可能な規模
- 自分が使いたい機能なので継続しやすい
- 必要なツール・ライブラリは無料で入手可能
```

### 8. 完了通知

ドキュメントのリンクを提示し、完了を通知する。

```text
アイディア深掘りが完了しました。以下のドキュメントをご確認ください。

[プロジェクトアイディアドキュメントを確認](computer:///mnt/user-data/outputs/{プロジェクト名}_brainstorm.md)
```
